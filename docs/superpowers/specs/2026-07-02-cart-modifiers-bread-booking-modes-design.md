# Спек: модификаторы в корзине/Telegram, хлеб как отдельная категория, два режима брони

Дата: 2026-07-02
Ветка: `feature/iiko-menu-phase1a`
Статус: черновик к реализации

## Контекст

Сайт ресторана «Кучер и Конга» (Next.js 16). Меню и бизнес-ланчи приходят из iiko
как позиции с группами модификаторов. Пользователь сообщил о четырёх проблемах,
объединённых общей причиной (структурные модификаторы нигде не отображаются):

1. При добавлении позиций в корзину не видно их модификаторы, в том числе не виден
   состав бизнес-ланчей (критично).
2. В бизнес-ланчах «Хлеб б/л» лежит внутри одного single-select-набора вместе с
   напитками → можно выбрать **либо** хлеб **либо** один напиток. Это нарушает
   бизнес-логику. Хлеб должен быть отдельной категорией: по умолчанию «с хлебом»
   (показан выбранным), либо «без хлеба» (в заказе не отображается). На цену не влияет.
3. В Telegram не приходят модификаторы блюд.
4. Форму брони нужно упростить: по умолчанию режим «Связаться с администратором»
   (только зал + контактные поля + пожелания), плюс режим «Выберу сам» с выбором
   типа заказа. Подтверждение в любом случае через администратора.

VK-уведомления (#5) и кнопка «Отправить в iiko» (#6) — **вне этого спека**, отдельные
циклы, требуют внешних доступов.

## Что уже есть в коде (проверено)

- `BusinessLunchConstructor.tsx` (публичный путь бизнес-ланчей, `EnhancedMenuSection`
  рендерит его при `selectedMenuType === 'business'`) **уже** пишет
  `modifiers: { group, option }[]` в позицию корзины (строка 54) и `description` с составом.
- `FoodDetailModal.tsx` (обычные блюда) структурные модификаторы **не** пишет —
  вклеивает их в `name` как `«Стейк (Пюре, Соус)»`; идентичность позиции держится на
  `id = item.id__modsKey`.
- `CartDrawer.tsx` `CartItemComponent` рендерит **только** `name` + `price` — ни
  `modifiers`, ни `description`. Поэтому даже уже сохранённый состав бизнес-ланча не виден.
- `lib/booking/formatTelegram.ts` (бронь/предзаказ) и `app/api/telegram/route.ts`
  (доставка) выводят по позиции только `name`/`qty`/`price`.
- Бизнес-ланчи приходят из iiko; `lib/iiko/mapMenu.ts` `mapModifierGroups` маппит
  `itemModifierGroups` один-в-один. Группа «хлеб + напитки» — single-select
  (`min:1, max:1`), внутри «Хлеб б/л» + чай/кофе/компот → отсюда баг #2.
- Форма брони в `app/HomeClient.tsx`: зал, имя, фамилия, телефон, дата, время,
  взрослых, детей, `BookingTypeSelector` (onsite/preorder/banquet), поле
  «Пожелания (необязательно)» **уже есть** (строка 1055, в state — `comment`).

## Дизайн

### Единая форма модификаторов (основа #1 и #3)

Тип `CartItem` в `types/index.ts` получает поле, совпадающее с тем, что уже эмитит
`BusinessLunchConstructor`:

```ts
modifiers?: { group: string; option: string }[];
```

**`FoodDetailModal.handleAdd`** (обычные блюда, ветка без `variant`):
- собрать `modifiers` из выбранных опций в форме `{ group: g.name, option: cleanOptName(opt.name) }`;
- `name` — базовое имя блюда (без склейки `(…)`); идентичность/дедуп остаются на
  существующем `cartId = item.id__modsKey`, поэтому поведение корзины не меняется.

**`CartDrawer.tsx` `CartItemComponent`**: под названием позиции вывести список
`item.modifiers` мелким текстом (`группа: опция`), если он не пуст. Одно место —
чинит и обычные блюда, и бизнес-ланчи.

**Telegram** (обе точки):
- `formatTelegram.ts` — в блоке предзаказа под каждой позицией вывести её модификаторы
  под-пунктами.
- `app/api/telegram/route.ts` — то же для доставки; тип `OrderItem` расширить
  `modifiers?: { group: string; option: string }[]`.
- В обеих точках отправки (сабмит брони и сабмит доставки в клиенте) при маппинге
  корзины в payload-позиции **прокидывать `item.modifiers`** (сейчас берутся только
  name/qty/price).

Показ модификаторов фильтрует служебную опцию «Без хлеба» (см. ниже), чтобы «без хлеба»
не выводилось.

### Хлеб отдельной категорией (#2)

Правка в `lib/iiko/mapMenu.ts` `mapModifierGroups`. Когда в группе есть опция-хлеб
(имя матчит `/хлеб/i`), разделить её на две группы:

- **«Хлеб»**: `{ min: 1, max: 1, options: [ { id: <реальный id хлеба>, name: 'С хлебом', price: 0 }, { id: 'no-bread', name: 'Без хлеба', price: 0 } ] }`.
  Порядок опций фиксирован: «С хлебом» первой (дефолт).
- **напитки**: та же группа без опции-хлеба (остальные опции, `min/max` исходные).

Если группа состояла только из хлеба — просто переименовать/нормализовать в «Хлеб» с
парой «С хлебом / Без хлеба».

`// ponytail:` комментарий помечает потолок эвристики — матч по `/хлеб/i`; апгрейд-путь:
явный id/конфиг, если iiko переименует группу.

**`BusinessLunchConstructor.tsx`**: инициализировать `choices` дефолтом для группы
хлеба (первая опция «С хлебом»), чтобы она была выбрана по умолчанию; гость может
переключить на «Без хлеба». Дефолт ставится только для группы, чьё имя матчит `/хлеб/i`
(не для напитков — их гость выбирает сам). `min:1` → хлеб всегда в составе, но дефолт
его удовлетворяет, поэтому кнопка «Добавить» не блокируется.

**Отображение**: при формировании списка модификаторов (корзина и Telegram)
отфильтровывать записи с опцией «Без хлеба» (id `no-bread`). Итог: «с хлебом» →
строка «Хлеб: С хлебом» видна; «без хлеба» → строка отсутствует. Цена не меняется в
обоих случаях (опции price 0).

### Два режима брони (#4)

Над формой брони в `app/HomeClient.tsx` — переключатель режима (новый локальный state,
дефолт `admin`):

- **«Связаться с администратором» (по умолчанию)**: зал, имя, фамилия, телефон, дата,
  время, взрослых, детей, пожелания. `BookingTypeSelector` и блоки предзаказа/банкета
  скрыты.
- **«Выберу сам»**: дополнительно показывается существующий `BookingTypeSelector`
  (заказ по факту / предзаказ / банкетное меню) и его зависимые блоки (сводка предзаказа,
  выбор банкетного пакета).

Обе ветки отправляются через тот же путь `/api/telegram` (`type: 'booking'`).
Подтверждение всегда через администратора (текст формы это уже сообщает). В payload
добавляется поле `mode: 'admin' | 'self'`; в сообщении Telegram при `mode === 'admin'`
выводится «Режим: Связаться с администратором», иначе — выбранный тип
(`formatTelegram.ts`). Поле «Пожелания» уже существует — просто должно быть видимым в
обоих режимах.

В режиме `admin` тип брони в payload можно не считать значимым; формат сообщения
ориентируется на `mode`.
