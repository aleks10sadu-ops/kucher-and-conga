# Бронь как заявка с авто-валидацией — дизайн (подпроект B)

**Дата:** 2026-06-27
**Проект:** Kongotest (`kucher-conga-website`, Next.js 16 / App Router, Vercel)
**Статус:** дизайн одобрен пользователем; на ревью спека

## Цель

Переработать бронирование столов: вместо проверки доступности через CRM —
**простая заявка**, которую администратор подтверждает звонком. Гость указывает
ФИО, телефон, дату/время, число взрослых и детей, выбирает **тип брони**
(по факту / предзаказ / банкет) и получает **подсказки и валидацию прямо на этапе
выбора** (а не после отправки). Текущая карточка брони сохраняется и дорабатывается.

## Текущее состояние (что меняем)

- Форма брони — в `app/page.tsx` (~1805 строк), поля `BookingData`:
  `firstName, lastName, phone, date, time, guests (одно число), comment, hallId` + согласие.
- Сабмит: `lib/reservations.ts` → CRM Supabase RPC `create_public_reservation`
  (+ дубль в `POST /api/telegram`).
- Проверка доступности: `hooks/useHallAvailability.ts` (CRM RPC `get_hall_month_availability`),
  вейтлист-гейт в `submitBooking`, жёлтая подсветка «занято» в `DateTimePicker`.
- Залы: `HallSelector.tsx` (CRM `halls` + локальный контент). Conga и залы Кучера
  различимы по имени. Залы: Conga, Морской (Кучер), Барный (Кучер), Веранда (Кучер),
  Летняя веранда, Банкетные залы, Беседки.
- Время: `lib/hooks/useBookingLogic.ts` (`isBookingTimeValid` — рабочие часы, мин. запас).

**Убираем:** `useHallAvailability` и его вызов, вейтлист-`window.confirm`, жёлтую
подсветку «занято» и легенду листа ожидания в `DateTimePicker`.
**Оставляем:** проверку рабочих часов (`isBookingTimeValid`), ограничения дат
администратора (`restricted_dates`/`restricted_times`), формат телефона, согласие ПД.

## Поля формы (итог)

`Фамилия`, `Имя`, `телефон`, `дата`, `время`, **`взрослые` (≥1)**, **`дети` (≥0)**,
`зал`, **`тип брони`** (`onsite` по факту / `preorder` предзаказ / `banquet` банкет),
`комментарий`, согласие. Одно поле `guests` заменяется на `adults`+`children`.
Условные поля: предзаказ → состав общей корзины сайта; банкет → выбранный пакет.

## Правила доступности типов

Отключаем недопустимые типы (двухуровневая модель: тип нельзя выбрать — показываем причину):

| Взрослых | По факту (`onsite`) | Предзаказ (`preorder`) | Банкет (`banquet`) |
|---|---|---|---|
| < 6  | ✅ | ✅ (сроки ниже) | ❌ — банкет от 6 взрослых |
| 6–8  | ✅ | ✅ | ✅ (дата ≥ +2 дня) |
| 9–11 | ❌ — от 9 взрослых только предзаказ или банкет | ✅ | ✅ (≥ +2 дня) |
| ≥ 12 | ❌ — от 12 взрослых только банкет | ❌ — от 12 взрослых только банкет | ✅ (≥ +2 дня) |

- **Дети** записываются отдельно и **не влияют** на пороги 6/9/12 (считаем только взрослых).
- Если для выбранного числа взрослых и срока **ни один тип недоступен**
  (напр. 12+ на завтра) → отправка заблокирована, показываем
  «Онлайн-заявка для такого числа гостей и срока недоступна — позвоните администратору».

## Сроки (по часовому поясу ресторана)

Все вычисления дат/времени — в фиксированном поясе ресторана `Europe/Moscow`
(константа `RESTAURANT_TZ`), не зависят от устройства гостя.

- **Предзаказ:** дата мероприятия ≥ завтра (день-в-день нельзя) **и** момент отправки
  не позже **16:00 дня до** мероприятия (т.е. `now ≤ 16:00 локального дня (eventDate − 1)`).
- **Банкет:** дата мероприятия ≥ **+2 дня** (сегодня и завтра — нельзя).
- **Время брони:** в рабочих часах ресторана (как сейчас, `isBookingTimeValid`).

## Залы: классификация и минимумы предзаказа

`classifyHall(name): 'conga' | 'kucher' | 'other'`:
- `conga` — имя содержит «Conga»;
- `other` — имя содержит «банкет» или «беседк» (Банкетные залы, Беседки);
- `kucher` — всё остальное (Морской/Барный/Веранда (Кучер), Летняя веранда).

**Минимум предзаказа (жёстко, по сумме еды в корзине):**
- `conga` → **4000 ₽**; `kucher` → **3000 ₽**; `other` → минимума нет → «свяжется администратор».
- Ниже минимума (для conga/kucher) → отправка заблокирована, подсказка «Доберите ещё N ₽».

## Предзаказ

- Источник позиций — **общая корзина сайта** (`useCart`). При типе «предзаказ» форма
  показывает состав корзины и сумму еды; пустая корзина → блок.
- **Подсказка при выборе «предзаказ»:** «Предзаказ оформляется набором блюд в корзину
  на сайте — наберите позиции, и заявка с их составом уйдёт администраторам на
  рассмотрение.» (показывается сразу при выборе типа, до отправки).
- При «по факту» корзина к заявке не прикрепляется.

## Банкет

- **Зал обязателен**, выбирается до пакета. Пакеты фильтруются по залу:
  - зал `conga` → только пакеты Conga (7500 / 6000 ₽/чел);
  - любой **не-conga** зал (kucher, Банкетные залы, Беседки) → **все** пакеты
    (Conga 7500/6000 + Кучер 5000).
- Источник пакетов — доработанная `BanquetMenuModal.tsx` (сейчас контент захардкожен):
  делаем пакеты **выбираемыми** (выбор → прикрепляется к заявке).
- Банкет всегда сопровождается инфо «Свяжется администратор для предоплаты и деталей».
  Отправка заявки разрешена (это заявка, не финальное бронирование).

## Чистый модуль правил `lib/booking/rules.ts`

Вся логика — здесь, изолированно и тестируемо. Форма только отображает результат.

```ts
export type BookingType = 'onsite' | 'preorder' | 'banquet';
export type HallGroup = 'conga' | 'kucher' | 'other';

export interface BookingRuleInput {
  adults: number;
  children: number;
  eventDate: string;   // 'YYYY-MM-DD'
  eventTime: string;   // 'HH:mm'
  now: Date;           // текущий момент
  hallGroup: HallGroup | null;
  type: BookingType | null;
  cartFoodSum: number; // ₽, сумма еды в корзине (0 если не предзаказ)
}

export interface TypeAvailability {
  type: BookingType;
  allowed: boolean;
  reason?: string;     // причина отключения (подсказка)
}

export interface BookingValidation {
  availableTypes: TypeAvailability[]; // все три, с allowed/reason
  canSubmit: boolean;
  blocking: string[];  // сообщения, блокирующие отправку
  info: string[];      // контекстные уведомления (админ свяжется, предоплата, подсказка предзаказа)
}

export function evaluateBooking(input: BookingRuleInput): BookingValidation;
export function classifyHall(hallName: string | null | undefined): HallGroup | null;
export function preorderMinimum(group: HallGroup): number | null; // 4000 | 3000 | null
export function banquetPackagesForHall(group: HallGroup | null): 'conga' | 'all' | null;
```

Тип по умолчанию — `onsite` (если доступен), иначе первый доступный.

## Сабмит

- Удаляем проверку доступности и вейтлист. `status` всегда `'new'`.
- CRM RPC `create_public_reservation` (как сейчас), с маппингом новых полей:
  - `p_guests_count` = `adults + children`;
  - `p_menu_type` = тип брони (`onsite` / `preorder` / `banquet`);
  - `p_comments` = структурный свод (взрослые/дети, тип, зал, для предзаказа — состав
    и сумма корзины, для банкета — выбранный пакет, исходный комментарий гостя).
  - (CRM RPC не принимает отдельный `children_count` — дети идут в `guests_count` и в свод.)
- Telegram (`POST /api/telegram`) — полный структурный состав заявки: ФИО, телефон,
  дата/время, взрослые/дети, зал, тип, состав предзаказа+сумма / банкетный пакет, комментарий.
- Сообщение об успехе: «Заявка принята! Администратор свяжется с вами для подтверждения.»

## Информирование на этапе выбора (UX)

При изменении взрослых/даты/зала/типа форма мгновенно:
- включает/отключает опции типов с причиной (из `availableTypes`);
- рядом с выбранным типом показывает контекстные сообщения (`info`/`blocking`):
  подсказку предзаказа, минимум и «доберите N ₽», «свяжется администратор для предоплаты»,
  срок банкета, случай «позвоните администратору».
Никакие из этих сообщений не появляются только после отправки.

## Тестирование

- Юнит-тесты чистого `lib/booking/rules.ts` (vitest, уже настроен): все ячейки таблицы
  (взрослые × тип), границы — 6/9/12 взрослых; отсечка 16:00 дня до (до/после);
  банкет +2 дня (сегодня/завтра/послезавтра); минимумы 4000/3000 и «прочие» залы;
  пустой случай (12+ на завтра → нет типов); фильтр банкетных пакетов по залу;
  `classifyHall` по всем известным именам залов.
- Ручная проверка формы: смена взрослых/даты/зала корректно меняет доступные типы и
  подсказки; предзаказ ниже минимума блокируется; банкет показывает нужные пакеты.

## Вне scope (YAGNI)

- Онлайн-оплата/предоплата (организует администратор).
- Проверка вместимости зала.
- Изменения в самой CRM (схема/RPC) — работаем в рамках текущей сигнатуры `create_public_reservation`.
- Подпроекты A (вкладки меню + модификаторы) и C (дизайн-проход) — отдельными циклами.
