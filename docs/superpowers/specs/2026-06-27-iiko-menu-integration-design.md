# Интеграция меню iiko — Фаза 1 (чтение меню)

**Дата:** 2026-06-27
**Проект:** Kongotest (`kucher-conga-website`, Next.js 16 / App Router, деплой Vercel, данные Supabase)
**Статус:** дизайн на согласовании

## Цель

Сделать **iiko единственным источником меню** на сайте. Текущее меню в Supabase
(`menu_types → categories → dishes → dish_variants`) и его редактирование в админке
перестают использоваться для витрины/заказа. Данные о позициях, ценах, КБЖУ, картинках,
а в дальнейшем о стоп-листе и остатках порций, берутся из внешнего меню iiko.

Фаза 1 — **только чтение** (рендер меню). Отправка заказов в iiko — Фаза 2 (отдельный спек).

## Подтверждённый контекст iiko (по факту, discovery 2026-06-27)

- API: `https://api-ru.iiko.services` (iiko Transport / Cloud API).
- Авторизация: `POST /api/1/access_token {apiLogin}` → `token` (~1 час), далее `Authorization: Bearer`.
- Организация: **«Кучер»**, `organizationId = 2418cf7b-1870-4e84-a181-4afc3d2d2506`.
- Внешнее меню одно: **«Яндекс»**, `externalMenuId = 79802`. Прайс-категорий нет.
- Получение меню: `POST /api/2/menu/by_id {externalMenuId, organizationIds, language:"ru"}`.
- Структура ответа (важные поля):
  - `itemCategories[]`: `{ id, name, items[] }` — **18 категорий**, всего **130 блюд**.
  - `items[]`: `{ itemId, sku, name, description, itemSizes[], type, isHidden }`.
  - `itemSizes[]` (у всех блюд **ровно один размер**):
    `{ sku, sizeName, portionWeightGrams, prices[].price, buttonImageUrl,
       nutritionPerHundredGrams{ fats, proteins, carbs, energy }, itemModifierGroups[] }`.
  - Цена: `itemSizes[0].prices[0].price` — целое число рублей (₽). У всех 130 блюд > 0.
  - Картинка: `itemSizes[0].buttonImageUrl` (у блюда `buttonImageUrl` обычно нет). 125/130 с картинкой.
  - КБЖУ: `itemSizes[0].nutritionPerHundredGrams` (на 100 г). Есть у **всех 130** блюд.
  - Описание: есть у 127/130.

### Бизнес-ланчи (сеты)

Категория **«БИЗНЕС ЛАНЧ»** = 4 фикс-сета (СЕТ №1–4, цены 580/520/470/430 ₽).
Каждый сет — обычное блюдо (`type: DISH`) с фиксированной ценой и набором
`itemSizes[0].itemModifierGroups[]`:

- группы: «Суп на сегодня», «Салат на сегодня», «Второе блюдо на сегодня»,
  «Гарниры на бизнес ланчи», «Выбор хлеба и напитков» (набор групп зависит от сета);
- у группы есть `restrictions { minQuantity, maxQuantity, byDefault }` (обычно max=1 — выбор одной опции);
- опции внутри группы имеют цену 0 (включены в стоимость сета), своё КБЖУ и вес.

То есть «конструктор бизнес-ланча» = выбрать сет → выбрать по одной опции в каждой группе.
Текущий `BusinessLunchBuilder` (на статике `businessLunchData`) перерабатывается под эту модель.

## Объём Фазы 1 и порядок инкрементов

Фаза 1 разбита на инкременты, каждый — отдельно проверяемый результат:

- **1A (первый видимый результат, делаем сейчас):** клиент iiko + маппер + рендер карточек
  блюд из iiko по всем категориям. Карточка показывает: **картинку, название, описание,
  цену, КБЖУ**. Источник меню для сайта переключается с Supabase на iiko.
- **1B:** переработка конструктора бизнес-ланчей под сеты iiko (модификаторные группы).
- **1C:** стоп-лист и остатки порций (`/api/1/stop_lists`, кэш ~60 с, `inStock`/`balance`,
  серый вид для нет-в-наличии, лимит порций).

Корзина/идентификаторы на iiko `itemId/sku` и **отправка заказов в iiko — Фаза 2**
(отдельный спек; зависит от маппинга из Фазы 1).

## Архитектура

Подход: **живой серверный фетч с кэшированием** на Vercel, без доп. инфраструктуры.

### Модуль `lib/iiko/` (изолированные единицы)

- `config.ts` — чтение и валидация env, типизированный конфиг.
  Env-переменные (Vercel + `.env.local`, **не коммитятся**):
  - `IIKO_API_LOGIN` — ключ (apiLogin).
  - `IIKO_ORGANIZATION_ID = 2418cf7b-1870-4e84-a181-4afc3d2d2506`.
  - `IIKO_EXTERNAL_MENU_ID = 79802`.
- `client.ts` — низкоуровневый `iikoPost(path, body, token?)`: базовый URL, JSON,
  обработка ошибок, один ретрай при 401 (после реавторизации).
- `auth.ts` — `getToken()`: токен в памяти модуля с TTL ~55 мин; повторная авторизация
  при истечении/401.
- `menu.ts` — `fetchExternalMenu()`: вызов `/api/2/menu/by_id`; кэш через
  `unstable_cache`, `revalidate: 300`, тег `iiko-menu`.
- `mapMenu.ts` — **чистая функция** `mapExternalMenu(raw): Record<string,{categories: MenuCategory[]}>`.
  Преобразует внешнее меню iiko в тот же формат, что сейчас отдаёт RPC `get_active_menu`,
  чтобы `EnhancedMenuSection` и другие потребители работали без переделки. На каждое блюдо:
  - `id` = `itemId`, `name`, `description`;
  - `price` = `itemSizes[0].prices[0].price`;
  - `weight` = `portionWeightGrams`;
  - `image` = `itemSizes[0].buttonImageUrl`;
  - `nutrition` = `nutritionPerHundredGrams` как есть, **на 100 г** (`energy` ккал,
    `proteins`/`fats`/`carbs` г), без пересчёта; `portionWeightGrams` показываем отдельно как вес порции;
  - `sku`, `categoryId`.
  - `isHidden`-блюда и блюда без активной цены отфильтровываются.
- `types.ts` — типы сырого ответа iiko (минимально нужные поля).

### Точки интеграции (минимум изменений UI)

- `app/actions/getMenu.ts → getMenuData()`: вместо Supabase RPC вызывает
  `fetchExternalMenu()` + `mapExternalMenu()`. Формат ответа прежний → `EnhancedMenuSection`
  (главная) работает без переделки.
- `lib/menu/getMenuByType.ts`: источник — модуль iiko (тот же маппер).
- `app/menu/page.tsx` (сейчас статика `app/data/*`): переводится на серверное получение
  из модуля iiko через `getMenuByType`/`getMenuData`.
- **КБЖУ в карточке:** тип `MenuItem` дополняется опциональным `nutrition`; карточки
  (`MenuItem.tsx`, `FoodDetailModal.tsx`, рендер в `app/menu/page.tsx`) показывают
  ккал/Б/Ж/У **на 100 г** (как в iiko), вес порции — отдельной строкой.

### Discovery-роут (одноразовый, dev)

`app/api/iiko/discover/route.ts` — защищён секретом (`?secret=` из env), возвращает
организации, внешние меню, прайс-категории и терминальные группы. Нужен для проверки/будущих
изменений и для Фазы 2 (терминальные группы). В прод-навигации не используется.

## Кэширование и свежесть

- Токен: ~55 мин в памяти модуля.
- Меню: `revalidate 300с` + тег `iiko-menu`. Доп. секретный роут
  `app/api/iiko/revalidate` (`revalidateTag('iiko-menu')`) — принудительно обновить
  после правок меню в iiko.
- Стоп-лист (инкремент 1C): отдельный кэш ~60с, мёрж на каждом запросе — свежесть
  стоп-листа не зависит от кэша меню.

## Обработка ошибок и устойчивость

- Каждый внешний вызов обёрнут; при сбое iiko отдаётся **последний удачный кэш**
  (stale-while-error). Если кэша ещё нет — пустое состояние с сообщением.
- 401 → одна реавторизация и повтор.
- На время Фазы 1 статические `app/data/*` могут остаться как аварийный фолбэк
  (как в текущем `getMenuByType`), позже удаляются.

## Категории и вкладки

Меню одно («Яндекс»). Текущая трёхвкладочная навигация на главной (`Кухня/Бар/Акции`)
и `menu_types` в Supabase **упраздняются для витрины**. Вкладок по типам нет: 18
`itemCategories` из iiko становятся разделами одного меню, навигация — по категориям
(якорная, как сейчас по категориям внутри вкладки). «БИЗНЕС ЛАНЧ» рендерится как
конструктор (1B), остальные 17 категорий — обычные карточки.

## Тестирование

- Юнит-тесты чистого `mapMenu.ts` на фикстуре (`scratchpad/menu.json` → фикстура в репо):
  расчёт КБЖУ на порцию, выбор цены/картинки, фильтрация `isHidden`, пустые описания.
- Dev-роут/скрипт для дампа смапленного меню и сверки счётчиков (18 категорий / 130 блюд).
- Ручная проверка: задать env → главная и `/menu` показывают карточки из iiko
  с картинкой/названием/описанием/ценой/КБЖУ; счётчики совпадают.

## Вне scope (YAGNI) — согласовано

- Отправка заказов в iiko (Фаза 2).
- Редактирование меню в админке (`/admin/dishes`, `/admin/categories`) — становится не нужно.
- Webhooks стоп-листа, сложные модификаторы/комбо вне бизнес-ланчей, прайс-категории.

## Безопасность

- `IIKO_API_LOGIN` хранится только в env (Vercel + `.env*.local`, уже в `.gitignore`).
  Ключ, присланный в чат, рекомендуется перевыпустить в iiko, так как он засветился.
- Все вызовы iiko — только на сервере (Server Components / Server Actions / Route Handlers),
  ключ и токен на клиент не попадают.
